# Telegram history import

## Summary

Импортировать доступную историю одного настроенного Telegram-чата в SignalForge как воспроизводимый набор исходных сообщений для последующей обработки.

## User value

Пользователь может один раз загрузить накопившуюся историю чата, включая существующие 900+ сообщений, не читая и не копируя их вручную.

## Scope

### In scope

- импорт доступных обычных сообщений с текстом или подписью к медиа из одного настроенного Telegram-чата;
- сохранение исходной идентичности, времени, автора и связей ответа сообщения, когда Telegram предоставляет эти данные;
- повторный безопасный запуск импорта без создания дубликатов;
- продолжение прерванного импорта;
- итоговый отчёт о результате импорта.

### Out of scope

- постоянное отслеживание новых сообщений после завершения импорта;
- загрузка и хранение файлов, изображений, аудио и видео;
- извлечение, проверка или классификация ссылок;
- группировка сообщений, LLM-анализ, поиск и генерация дайджестов;
- синхронизация последующих редактирований и удалений в Telegram;
- Telegram service messages, forwarded-from metadata и реакции;
- импорт нескольких чатов за один запуск;
- пользовательский интерфейс и Telegram-бот для управления импортом.

## Requirements

- **REQ-001:** Пользователь должен иметь возможность запустить импорт для одного предварительно настроенного Telegram-чата.
- **REQ-002:** Система должна обработать всю историю, доступную используемой Telegram-учётной записи на момент запуска, и импортировать каждое обычное сообщение, содержащее текст или подпись к медиа.
- **REQ-003:** Для каждого импортированного сообщения система должна сохранить идентификатор исходного чата, идентификатор сообщения, дату и время публикации, доступную информацию об авторе, текст или подпись к медиа, тип вложения при его наличии и идентификатор сообщения-родителя для ответа, если он доступен.
- **REQ-004:** Сообщение должно однозначно идентифицироваться сочетанием исходного чата и Telegram message ID.
- **REQ-005:** Повторный импорт одной и той же истории не должен создавать дубликаты сообщений.
- **REQ-006:** После прерывания импорт должен допускать повторный запуск и завершение без потери уже сохранённых сообщений и без дубликатов.
- **REQ-007:** Недоступные необязательные поля сообщения не должны останавливать импорт остальных сообщений.
- **REQ-008:** Ошибка доступа к чату или Telegram должна завершать запуск с явным сообщением об ошибке без выдачи частичного результата за успешный импорт.
- **REQ-009:** После запуска система должна сообщить итоговый статус и количество впервые сохранённых, ранее существовавших, пропущенных по правилам scope и не импортированных из-за ошибок сообщений.
- **REQ-010:** Telegram credentials и session data не должны попадать в исходный код, логи, отчёты импорта или хранилище сообщений.

## Acceptance criteria

- **AC-001** (covers REQ-001, REQ-002): Для доступного настроенного чата с конечной историей успешный запуск обрабатывает историю от самого раннего до самого позднего доступного сообщения, импортирует все обычные сообщения с текстом или подписью и завершается успешно.
- **AC-002** (covers REQ-003, REQ-004): Сохранённая запись текстового сообщения содержит исходные chat ID и message ID, timestamp, доступного автора и исходный текст; запись ответа дополнительно содержит исходный parent message ID.
- **AC-003** (covers REQ-003): Для медиа-сообщения с подписью сохраняются метаданные сообщения, подпись и тип вложения, но бинарный файл не загружается.
- **AC-004** (covers REQ-005): Два последовательных импорта неизменившейся истории оставляют по одной записи на пару chat ID/message ID; второй запуск сообщает ноль впервые сохранённых сообщений.
- **AC-005** (covers REQ-006): После прерывания на середине истории повторный запуск сохраняет оставшуюся историю, не дублируя уже записанную часть.
- **AC-006** (covers REQ-007): Сообщение с текстом, но без доступного автора сохраняется с пустым необязательным полем автора, а импорт продолжает обработку истории.
- **AC-007** (covers REQ-008): При отказе авторизации или доступа запуск завершается с неуспешным статусом и диагностикой без credentials/session data.
- **AC-008** (covers REQ-009): Итог успешного или частично успешного запуска содержит статус и счётчики новых, существовавших, пропущенных и ошибочных сообщений, сумма которых равна числу обработанных сообщений.
- **AC-009** (covers REQ-010): Автоматическая проверка логов, результата запуска и сохранённых message records не обнаруживает Telegram credentials или session data.

## Constraints

- Источником является один Telegram-чат и одна авторизованная учётная запись пользователя.
- Сообщения хранятся в PostgreSQL согласно утверждённой product specification.
- Конкретная Telegram-библиотека, схема данных и способ запуска определяются technical design.
- Импорт должен учитывать ограничения Telegram API и не обходить их.

## Edge cases and failure behavior

- Пустой доступный чат завершается успешно с нулевыми счётчиками.
- Отсутствующие автор или reply reference считаются допустимыми отсутствующими данными.
- Service messages и сообщения без текста или подписи пропускаются и не считаются ошибочными.
- Отдельное некорректное сообщение учитывается как ошибочное и не препятствует обработке оставшейся истории, если соединение и авторизация остаются рабочими.
- Потеря соединения, авторизации или доступа завершает текущий запуск как неуспешный; сохранённые до ошибки сообщения остаются доступны для повторного запуска.
- Нарушение уникальности исходного chat ID/message ID не создаёт вторую запись.

## Dependencies

- Утверждённая SignalForge product specification.
- Доступ пользователя к целевому Telegram-чату.
- Будущий technical design должен определить безопасную конфигурацию Telegram-доступа и PostgreSQL.
