# SignalForge: вводные и вопросы перед созданием SDD-инфраструктуры

## Цель текущего этапа

Подготовить SDD-инфраструктуру репозитория и reusable skills для Codex.

На этом этапе нельзя:

- создавать приложение или писать production-код;
- добавлять Python-код и зависимости;
- настраивать Docker Compose;
- создавать базу данных;
- реализовывать Telegram-интеграцию;
- обращаться к внешним сервисам без явного поручения.

## Контекст продукта

SignalForge — персональная система для анализа информационных чатов.

Первым источником данных планируется Telegram-чат об LLM, AI-native разработке и coding agents. В будущем система должна импортировать историю, отслеживать новые сообщения, извлекать полезные материалы, группировать обсуждения, создавать выжимки, формировать базу знаний и выпускать ежедневные и еженедельные дайджесты.

Эти возможности пока являются контекстом продукта, а не разрешением на реализацию.

## Зафиксированные правила процесса

1. Спецификации являются источником истины.
2. Поведение нельзя реализовывать, если оно отсутствует в утверждённой feature specification.
3. Codex не может самостоятельно присваивать спецификации статус `Approved`.
4. Реализация выполняется по одной задаче за раз.
5. Перед реализацией конкретной задачи Codex читает связанные `spec.md`, `design.md` и `tasks.md`.
6. Завершённость задачи подтверждается acceptance criteria и релевантными проверками.
7. Checkbox задачи обновляется только после выполнения её критериев.
8. Значимые технические решения фиксируются отдельно.
9. Проверка спецификации не должна автоматически исправлять код.

## Предлагаемый SDD-процесс

1. Создание или обновление product specification.
2. Создание feature specification со статусом `Draft`.
3. Review feature specification.
4. Явное утверждение спецификации человеком.
5. Создание technical design.
6. Декомпозиция design на задачи.
7. Реализация одной выбранной задачи.
8. Проверка задачи по её критериям.
9. Проверка всей feature против acceptance criteria.
10. Code review.
11. Обновление статусов спецификации и задач.

## Предлагаемая структура репозитория

```text
AGENTS.md
README.md

specs/
  README.md

  product/
    product-spec.md

  features/
    001-telegram-ingestion/
      spec.md
      design.md
      tasks.md
      decisions.md

  architecture/
    system-overview.md
    data-model.md
    security.md

  templates/
    product-spec-template.md
    feature-spec-template.md
    design-template.md
    tasks-template.md

.agents/
  skills/
    create-feature-spec/
      SKILL.md
    review-feature-spec/
      SKILL.md
    create-technical-design/
      SKILL.md
    decompose-feature-tasks/
      SKILL.md
    implement-spec-task/
      SKILL.md
    verify-spec/
      SKILL.md
```

Два skills сверх первоначального списка предложены для формализации обязательных этапов review и task decomposition.

## Предлагаемая граница документов

### `AGENTS.md`

Короткие постоянные правила работы в репозитории: SDD-процесс, approval, безопасность, одна задача за раз, проверки и definition of done. Подробные шаблоны и требования продукта в него не входят.

### Specifications

Источник истины о целях, требованиях и принятых решениях:

- product spec описывает продуктовые цели и границы;
- feature spec определяет наблюдаемое поведение;
- design описывает техническую реализацию утверждённой feature spec;
- tasks задаёт исполнимый план;
- decisions фиксирует значимые решения и альтернативы;
- architecture содержит устойчивые общесистемные ограничения.

### Skills

Повторяемые инструкции для Codex: что прочитать, какие предусловия проверить, какой артефакт подготовить, чего не делать и когда остановиться. Skills не должны дублировать требования SignalForge или шаблоны спецификаций.

## Вопросы, требующие решения

### 1. Состав skills

Подтвердить один из вариантов:

- оставить четыре первоначальных skills: `create-feature-spec`, `create-technical-design`, `implement-spec-task`, `verify-spec`;
- принять рекомендуемые шесть skills, добавив `review-feature-spec` и `decompose-feature-tasks`.

Рекомендуемый вариант: шесть skills, поскольку review и декомпозиция уже являются обязательными этапами процесса.

### 2. Статусы feature specification

Какой набор статусов использовать?

Предлагаемый вариант:

```text
Draft → In Review → Approved → Implemented → Verified
                    ↘ Rejected
```

Нужно решить:

- требуется ли отдельный статус `In Review`;
- требуется ли `Rejected` или достаточно вернуть документ в `Draft`;
- кто и когда устанавливает `Implemented` и `Verified`;
- должна ли существенная правка утверждённой spec автоматически возвращать её в `Draft`.

### 3. Формат человеческого approval

Предлагается хранить в YAML frontmatter каждой feature spec:

```yaml
status: Draft
approved_by:
approved_at:
```

Нужно определить:

- достаточно ли команды пользователя в диалоге для изменения статуса;
- какое значение записывать в `approved_by`;
- требуется ли дата в ISO 8601;
- должен ли approval фиксироваться также в `decisions.md` или достаточно frontmatter и истории Git.

### 4. Статус product specification

Нужно решить, применяется ли тот же approval-процесс к `product-spec.md`.

Рекомендация: product spec также имеет `Draft` и `Approved`, а feature spec нельзя утверждать, если она противоречит product spec или product spec ещё не утверждена.

### 5. Идентификаторы и трассируемость

Предлагается использовать стабильные ID:

- `REQ-001` — требование;
- `AC-001` — acceptance criterion;
- `TASK-001` — задача;
- `DEC-001` — решение.

Нужно подтвердить формат и решить, уникальны ли ID только внутри feature или во всём репозитории.

Рекомендация: `TASK` и `DEC` уникальны внутри feature; ссылки между feature используют префикс feature, например `001-telegram-ingestion/AC-003`.

### 6. Definition of Done для задачи

Предлагаемый минимум:

- реализован только scope выбранной задачи;
- выполнены связанные acceptance criteria;
- добавлены или обновлены релевантные тесты;
- пройдены применимые проверки;
- не добавлено поведение вне утверждённой spec;
- checkbox задачи обновлён;
- значимые решения зафиксированы;
- отсутствуют секреты и чувствительные данные.

Нужно подтвердить, достаточно ли этого набора.

### 7. Команды проверки

В `AGENTS.md` требуется указать команды проверки, но технологический стек ещё не выбран.

Нужно решить:

- оставить до выбора стека явный placeholder `TBD — commands must be defined before implementation`;
- либо пока указать только универсальные документные проверки, если они появятся;
- кто утверждает будущие команды lint, test, type-check и security-check.

Рекомендация: не придумывать команды до выбора стека; зафиксировать, что design первой реализуемой feature обязан определить их до task implementation.

### 8. Architecture-документы

Нужно решить, создавать ли сейчас пустые или минимальные:

- `system-overview.md`;
- `data-model.md`;
- `security.md`.

Рекомендация: создать их как короткие документы со статусом и границами ответственности, но не заполнять спекулятивными техническими решениями. Альтернатива — добавить только каталог и описать ожидаемые документы в `specs/README.md`, создавая сами файлы по мере появления утверждённых решений.

### 9. Первая feature

Нужно решить, создавать ли сейчас каталог `001-telegram-ingestion` как пример или оставить только templates.

Рекомендация: на этапе инфраструктуры не создавать feature-документы с выдуманным содержанием. Создать каталог первой feature позднее через `create-feature-spec`, когда будут предоставлены конкретные требования.

### 10. Политика изменения утверждённой spec

Предлагаемый порядок при существенном изменении требований:

1. Вернуть spec в `Draft`.
2. Зафиксировать причину изменения.
3. Провести impact analysis для design, tasks и существующей реализации.
4. Повторить review.
5. Получить новое человеческое approval.

Нужно определить, какие изменения считаются существенными. Рекомендация: любое изменение scope, requirement, constraint или acceptance criterion; исправления формулировок без изменения смысла approval не сбрасывают.

### 11. Task verification и feature verification

Нужно подтвердить два уровня проверки:

- после каждой задачи — локальная проверка task acceptance criteria;
- после завершения всех задач — `verify-spec` для всей feature.

Рекомендация: использовать оба уровня.

### 12. Формат `decisions.md`

Нужно выбрать:

- единый журнал решений внутри feature;
- отдельный файл на каждое решение.

Рекомендация для первого этапа: единый `decisions.md` с записями `DEC-001`, датой, контекстом, решением, альтернативами и последствиями. Отдельные ADR-файлы вводить только при росте проекта.

### 13. Облегчённый процесс для мелких изменений

Нужно подтвердить, что полный SDD-цикл не требуется для:

- исправления опечаток;
- форматирования;
- изменений документации без изменения поведения;
- технического обслуживания, не влияющего на продуктовые требования.

При любом изменении наблюдаемого поведения должна существовать утверждённая spec.

### 14. Язык документов

Нужно выбрать основной язык для `AGENTS.md`, specifications, templates и skills:

- русский;
- английский;
- смешанный вариант: инструкции на английском, продуктовые документы на русском.

Рекомендация: выбрать один основной язык для всех нормативных документов, оставив технические идентификаторы и устоявшиеся термины на английском.

### 15. Инициализация Git

Текущий каталог не является Git-репозиторием.

Нужно уточнить, входит ли `git init` в следующий этап. Это отдельное изменение состояния; без явного поручения оно выполняться не будет.

## Правила безопасности для будущей инфраструктуры

- Никогда не читать и не выводить `.env`.
- Никогда не выводить Telegram API credentials.
- Никогда не читать или выводить Telegram session.
- Никогда не коммитить секреты.
- Не выполнять destructive-команды.
- Не устанавливать глобальные зависимости.
- Не обращаться к внешним сервисам без явного поручения.
- Не включать реальные credentials или session data в примеры, fixtures и документацию.

## Предлагаемый следующий шаг

После ответов на вопросы выше можно одним отдельным этапом создать только согласованную SDD-инфраструктуру: `AGENTS.md`, `README.md`, структуру `specs/`, templates и instruction-only skills. Продуктовый код, зависимости и интеграции при этом не создаются.
