# SignalForge — Project Brief

## Purpose

SignalForge — персональная AI-native система, которая превращает поток информации в структурированную базу знаний.

Первая задача проекта — анализ Telegram-чата, посвящённого:

- LLM
- AI-native разработке
- Coding Agents
- MCP
- Prompt Engineering
- Context Engineering
- AI tooling

В дальнейшем проект должен поддерживать дополнительные источники информации (GitHub, RSS, Reddit, Hacker News, Slack и др.).

---

# Problem Statement

В тематических чатах ежедневно появляется большое количество полезной информации.

Основные проблемы:

- невозможно регулярно читать весь чат;
- полезные ссылки быстро теряются;
- спустя месяц невозможно вспомнить, где обсуждалась нужная тема;
- многие идеи повторяются;
- отсутствует единая база знаний.

Проект должен автоматически превращать поток сообщений в структурированную базу знаний.

---

# Vision

Из хаотичного информационного потока получать:

- краткие дайджесты;
- темы обсуждений;
- полезные ссылки;
- рекомендации;
- список инструментов;
- базу знаний;
- возможность поиска.

---

# MVP

Первая версия должна уметь:

1. Импортировать историю Telegram-чата.
2. Хранить сообщения в PostgreSQL.
3. Следить за новыми сообщениями.
4. Извлекать ссылки.
5. Группировать сообщения.
6. Выполнять LLM-анализ.
7. Генерировать дайджест.

---

# Out of Scope

На MVP НЕ реализуются:

- веб-интерфейс;
- многопользовательский режим;
- авторизация;
- публичный API;
- облачная инфраструктура;
- обработка нескольких чатов одновременно;
- интеграция с Notion;
- интеграция с Obsidian.

---

# Development Methodology

Проект разрабатывается по Spec-Driven Development (SDD).

Любое изменение должно начинаться со спецификации.

Последовательность разработки:

1. Product Spec
2. Feature Spec
3. Technical Design
4. Task Decomposition
5. Implementation
6. Verification
7. Review

Код не должен реализовывать функциональность, отсутствующую в утверждённой спецификации.

# AI Workflow

При работе Codex обязан придерживаться следующего процесса:

1. Найти соответствующую feature specification.
2. Проверить её статус.
3. Прочитать design.
4. Найти task.
5. Реализовать только выбранную задачу.
6. Добавить тесты.
7. Выполнить проверки.
8. Обновить статус задачи.

Запрещено самостоятельно переходить к следующему этапу разработки.

---

# Coding Principles

Проект придерживается следующих принципов.

## Простота

Предпочитать простые решения сложным.

## Итеративность

Каждая задача должна быть небольшой.

## Минимальные зависимости

Не добавлять новые библиотеки без необходимости.

## Чистая архитектура

Бизнес-логика должна быть максимально независимой.

## Явные интерфейсы

Все взаимодействия между компонентами должны быть явно определены.

---

# Quality

Каждая завершённая задача должна:

- проходить тесты;
- соответствовать acceptance criteria;
- иметь понятные типы;
- не нарушать существующую архитектуру.

---

# Security

Никогда не:

- читать `.env`;
- выводить секреты;
- логировать Telegram API credentials;
- логировать Telegram Session;
- коммитить session-файлы.

# Success Criteria

Проект считается успешным, если пользователь сможет:

- перестать читать весь Telegram-чат вручную;
- получать краткие ежедневные дайджесты;
- быстро находить полезные материалы;
- иметь постоянно пополняемую базу знаний;
- доверять качеству автоматически созданных выжимок.
